# 子任务进度条（TaskCard）设计文档

日期：2026-07-09
负责范围：主页日历下方任务列表卡片（`TaskCard`）新增每个子任务的进度条展示。

## 背景

主页需求增补第 2 条要求"日历样式，能够显示每个子任务的进度条"。经过代码探查确认：

- 周日历条（`WeekCalendarComponent.ets`）每天格子只显示一个汇总三态图标（未开始/部分完成/全部完成），来自 `dayInfo.finTaskNum/targetTaskNum`，格子宽度仅 `100/7%`，不适合塞入 6 条子任务进度条。
- 点选某一天后，日历下方已有"任务列表"区域（`HomeComponent.ets` 渲染，逐个 `TaskCard` 组件），展示当天 6 个健康任务（早起、喝水、吃苹果、微笑、刷牙、早睡）。目前每张卡片右侧只用文字显示 `finValue / targetValue`（如"4 / 6 杯"）或"已完成"，没有可视化进度条。
- 每个任务的进度数据（`TaskInfo.finValue` / `targetValue` / `isDone`）已经完整存在，选中某天后即从数据库加载到 `selectedDayInfo.taskList`，不需要改动数据层或数据库表结构。

经与需求方确认，本次改动范围**只涉及任务列表卡片**，不改动周日历条本身。

## 目标

在任务列表的每张 `TaskCard` 上新增一条可视化进度条，直观展示当前任务的完成程度，同时保留原有的文字展示（"已完成" / "4 / 6 杯"）。

## 非目标

- 不改动周日历条 `WeekCalendarComponent.ets` 的三态图标展示。
- 不改动数据库表结构、`TaskInfo` 数据模型、打卡逻辑（`taskClock` / `updateTask`）。
- 不涉及"刷牙、微笑目标可设置 1~3 次"（主页需求增补第 1 条，属于另一位同学负责范围），但设计需保证该改动落地后无需再调整本设计的进度计算公式。

## 设计

### 组件改动

**`TaskCardComponent.ets`**：卡片内部结构由单行 `Row` 改为两行 `Column`：

```
🌞 早起              已完成
   ████████████████████████   ← 新增第二行：胶囊型进度条
```

- 第一行：保持现状不变（左侧图标 + 任务名，右侧 `finValue / targetValue` 文字或"已完成"）。
- 第二行：新增 ArkUI 原生 `Progress` 组件，`type: ProgressType.Capsule`，与代码库中已有的 `progress/pages/ProgressCard.ets` 里的 `Progress` 用法保持技术栈一致（该组件当前用 `ProgressType.Ring`，本设计用 `Capsule` 因为是横向卡片场景）。
- 进度条填充色使用现有色值 `blueColor`（`#007DFF`），轨道底色使用现有色值 `progress_background_color`（`#08000000`），与首页主色调一致，不新增颜色资源。
- 进度条宽度撑满卡片可用宽度，高度约 6vp（沿用代码库现有 dimen 命名习惯新增资源项，而非硬编码数字）。

**`HomeComponent.ets`**：`ForEach` 渲染 `TaskCard` 处目前写死 `.height($r('app.float.default_64'))`。因卡片改为两行需要更高的高度，在 `resources/base/element/dimen.json` 新增 `default_88`（沿用现有 `default_64/66/71/72` 的命名与递增风格），并把引用换成新增的资源项。

### 进度值计算

计算逻辑放在 `TaskCard` 内部（不改数据层），公式：

```
isDone === true  → 进度 = 100%
isDone === false → 进度 = finValue / targetValue 的百分比；
                    若 targetValue 不是有效数字（如"08:00"这种时间字符串），
                    或计算结果非数字，兜底为 0%
                    结果对 100% 取上限（min 100）
```

这个公式不需要按任务类型写分支，天然覆盖当前 6 个任务的两种形态：

- **打卡即完成型**（早起、早睡）：`targetValue` 是时间字符串（如`"08:00"`），无法转成数字，公式兜底为 0%；一旦 `isDone` 变为 `true`，进度直接跳到 100%。即这两个任务的进度条只有 0% / 100% 两种状态，没有中间态——这是符合其业务含义的预期行为，不是缺陷。
- **数值累计型**（喝水、吃苹果，以及未来可能调整为可累计的微笑、刷牙）：`finValue` 随打卡逐步增长，进度条会展示中间态（如 4/6 → 67%）。该公式无需等其他同学的"1~3 次"改动落地后再调整。

### 交互

- 进度条为纯展示，不可拖动，不改变现有点击（进入打卡弹窗）与长按（跳转编辑页）手势逻辑。

## 影响范围与风险

- 卡片变高后，任务列表整体滚动区域变长，视觉上更占空间，但不影响现有滚动逻辑（`Scroll` 组件已有 `onScroll`/`scrollTo` 逻辑，行数增多不改变其行为）。
- 新增 dimen 资源项属于纯增量修改，不影响其他引用了 `default_64` 的地方（需在改动时确认没有其他组件复用同一 `TaskCard` 高度设置；目前搜索确认 `TaskCard` 仅在 `HomeComponent.ets` 一处被引用）。

## 测试计划

- 手动验证：早起/早睡任务打卡前后进度条从 0% 跳到 100%；喝水/吃苹果多次打卡后进度条逐步增长直至 100%；未打卡的任务进度条为空。
- 手动验证：卡片高度变化后任务列表滚动、点击打卡弹窗、长按编辑跳转均正常。
