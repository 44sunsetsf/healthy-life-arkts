# AI 智能助手模块技术文档

## 一、技术栈

| 层级 | 技术/库 | 说明 |
|---|---|---|
| 开发框架 | ArkTS (HarmonyOS NEXT) | 声明式 UI 框架，`@Component` + `@State` 状态驱动 |
| 语言 | TypeScript (ArkTS 方言) | 基于 TypeScript 扩展，支持鸿蒙 API |
| 大模型 API | DeepSeek Chat (`deepseek-chat`) | 通过 HTTP POST 调用 `/v1/chat/completions` |
| 网络通信 | `@ohos.net.http` | 鸿蒙原生 HTTP 模块 |
| 数据持久化 | `@ohos.data.relationalStore` (RDB) | 鸿蒙关系型数据库，存储聊天记录 |
| 构建工具 | hvigor | 鸿蒙官方构建系统 |
| 通信协议 | RESTful API (JSON) | 请求/响应均为 JSON 格式 |
| UI 组件 | 声明式组件 (TextInput, Scroll, Column, Row, Stack 等) | 鸿蒙 ArkUI 原生组件 |

## 二、模块架构

```
ai/
├── model/
│   └── ChatMessage.ets          # 数据模型：消息角色枚举、消息类、API 接口定义
├── pages/

├── service/

common/database/tables/

```

## 三、功能清单

### 1. 基于当天打卡数据的智能分析

每次发送消息时，系统自动读取用户当天的任务完成情况（早起、喝水、吃苹果、每日微笑、每日刷牙、早睡共 6 项），构建为健康上下文，注入 System Prompt 发送给 DeepSeek 大模型。AI 能根据用户的真实打卡数据提供个性化建议。

- 示例上下文：`今日打卡：早起 ✅、喝水 ❌（2/8杯）、…总体进度：3/6已完成`
- 实现文件：`AiService.ets:38-57`、`AiChatPage.ets:137-170`

### 2. 欢迎消息

首次进入 AI 助手页面自动显示欢迎语：*"你好！我是你的健康生活AI助手。你可以问我关于健康习惯、饮食、运动等方面的问题…"*

- 欢迎消息位于页面顶部，不会被垂直居中
- 实现文件：`AiChatPage.ets:41-45`

### 3. 悬浮窗可拖动功能

应用内有一个粉色猫猫悬浮按钮，支持：
- **拖动**：手指按住可任意拖动到屏幕任意位置
- **点击**：点击后自动切换到 AI 助手 Tab 页
- **拖动/点击区分**：移动超过 5px 视为拖动，否则视为点击

- 实现文件：`AiFloatBtn.ets`

### 4. 历史记录功能

所有对话持久化存储到鸿蒙 RDB 数据库中：

- **存储内容**：每条消息的 `role`（user/assistant/system）、`content`、`timestamp`
- **会话分组**：按 SYSTEM 消息作为分隔符，将对话自动分组为不同会话，每组显示首条用户消息作为标题、会话日期、消息条数
- **恢复历史**：点击历史记录中的会话，可完整恢复该会话的所有消息到当前界面
- **新建对话**：点击"＋"按钮插入 SYSTEM 分隔标记并开启新会话

- 实现文件：`ChatMsgApi.ets`、`AiChatPage.ets:55-129`

### 5. 用户/AI 气泡样式区分

- **用户消息**：紫色圆角气泡（`#6C5CE7`），右对齐，带阴影
- **AI 消息**：白色圆角气泡，左侧有 🤖 机器人头像，左对齐，带阴影
- 气泡圆角采用非对称设计（用户右下角 4vp，AI 左上角 4vp）

- 实现文件：`AiChatPage.ets:292-351`

### 6. Loading 加载动画

等待 AI 回复时显示加载占位气泡，包含三个跳动圆点（`• • •`），右侧放置 🤖 头像。

- 实现文件：`AiChatPage.ets:354-380`

### 7. Markdown 格式处理

DeepSeek API 返回的内容可能包含 Markdown 加粗（`**text**`）和斜体（`*text*`）标记，客户端自动去除这些标记，确保纯文本显示。

- 实现文件：`AiService.ets:96-97`

### 8. 新消息自动滚动到底部

每次新增消息（用户发送 / AI 回复）和新建对话时，自动将消息列表滚动至最底部，确保最新消息始终可见。

- 实现文件：`AiChatPage.ets:198-202`

### 9. 网络错误处理

API 调用失败时（网络异常、服务端错误等），自动展示错误提示消息："抱歉，网络请求失败了，请检查网络连接和API配置后重试。"，防止 UI 崩溃。

- 实现文件：`AiChatPage.ets:189-191`

### 10. API 可配置

支持运行时动态修改 API 地址、API Key、模型名称，无需重新编译。

- `configure(apiUrl, apiKey, model?)` 方法
- 实现文件：`AiService.ets:29-36`

### 11. Tab 导航集成

AI 助手作为底部 Tab 栏的第 4 个标签页（共 5 个：首页、健康、成就、AI、我的），集成到主页面导航中。

- 悬浮按钮通过 `GlobalContext.setObject('switchToAiTab', callback)` 注册回调，实现跨组件 Tab 切换
- 实现文件：`MainPage.ets:52-53`

### 12. 输入框交互

- **占位提示**："输入你的问题..."
- **发送方式**：点击发送按钮（➤）或键盘回车
- **发送后清空**：消息发送后输入框文字自动清空
- **加载中防重复发送**：AI 回复未返回前，输入框可编辑但发送按钮不响应

- 实现文件：`AiChatPage.ets:172-196`

### 13. 数据库初始化

应用启动时自动创建 `chatMsg` 表，包含 `id`（自增主键）、`role`、`content`、`timestamp` 四个字段。

- 实现文件：`EntryAbility.ets:99-104`

## 四、痛点分析

### 1. ⚠️ API Key 硬编码在前端

`AiService.ets:26` 中 DeepSeek API Key 以明文形式硬编码在客户端代码中。任何人反编译应用即可获取密钥，存在被盗用、配额耗尽、账单损失等严重安全风险。

**建议**：将 API Key 移至后端服务，由后端代理转发 LLM 请求；或至少使用鸿蒙凭据管理服务加密存储，并在服务端做速率限制。

### 2. ❌ 不支持流式输出（SSE）

当前采用普通 HTTP POST 请求——等待完整回复返回后再一次性渲染。对于长回复，用户会看到较长时间的 loading 动画，体验不佳。

**建议**：对接 DeepSeek 的 streaming 模式（`stream: true`），使用 SSE 逐字渲染回复内容。

### 3. 🔇 缺少对话删除能力

数据库层只提供了 `deleteAll()`（清空全部），没有按会话删除或删除单条消息的能力。用户无法清理不需要的历史记录。

**建议**：新增按时间戳范围删除（对应会话删除）和按 id 删除的功能。

### 4. ⏱️ 消息同步阻塞用户体验

`sendMessage()` 是 async/await 模式，虽不阻塞主线程，但没有超时外的取消机制。如果 AI 回复过慢（readTimeout 60 秒），用户只能等待，无法中途取消。

**建议**：添加取消按钮，使用 `AbortController` 或手动 destroy http 请求。

### 5. 📝 System Prompt 硬编码且单一

系统提示词固定不变，仅包含健康数据上下文，无法根据不同场景（饮食咨询、运动建议、心理疏导等）切换角色设定。

**建议**：将 System Prompt 模板化，支持从配置文件或远程读取，并支持多场景切换。

### 6. 🔄 无本地缓存 / 离线降级

完全依赖网络和 DeepSeek API。网络不可用时，AI 助手完全不可用，且没有任何本地兜底回复或离线提示的预处理。

**建议**：预置一些通用的本地健康建议回复，网络不可用时作为降级方案。

### 7. 📦 历史记录分组逻辑脆弱

会话分组依赖 SYSTEM 角色消息作为分隔标记。如果 SYSTEM 消息插入失败或被误删，所有对话会合并为一个会话，分组逻辑崩溃。

**建议**：增加 `session_id` 字段，显式标记每条消息所属会话，替代隐式的 SYSTEM 分隔符方案。

### 8. 🔒 数据库操作无事务保护

消息插入和查询未使用事务，在高频发送场景下可能出现数据不一致（例如 SYSTEM 分隔消息插入失败但用户消息已写入）。

**建议**：相关联的多条写入操作包裹在数据库事务中。

### 9. 📐 悬浮按钮缺少位置边界限制和持久化

悬浮按钮位置仅在内存中记录（`@State posX/posY`），应用重启后恢复默认位置。且没有屏幕边界检测，可能被拖出可视区域。

**建议**：增加边界限制逻辑，并将位置持久化到本地存储。

### 10. 🔐 无用户身份隔离

当前所有聊天记录存储在同一个数据库表中，没有用户标识字段。如果未来支持多用户登录，所有用户的聊天记录会混杂在一起。

**建议**：在 `chatMsg` 表中增加 `user_id` 字段，查询时按用户过滤。

### 11. 🎨 无消息时间戳展示

虽然数据模型中有 `timestamp` 字段，每条消息也存储了时间戳，但聊天气泡 UI 中没有展示发送时间。

**建议**：在气泡下方或气泡内添加时间显示（如 `09:30`）。

### 12. 📊 缺少使用统计和分析

没有记录用户与 AI 的互动次数、对话时长、常见问题分类等数据，无法做进一步的用户行为分析和产品优化。

**建议**：埋点记录用户使用行为，用于后期优化 AI 回复质量和产品迭代。
